# Lanzou-Max 与蓝奏云交互分析

> 基于仓库现有代码整理，适合“对照源码 + curl 自己验证”使用。

## 1. 整体架构

- 项目没有直接暴露蓝奏云网页接口给前端，而是把蓝奏交互封装在 `lan/internal/lanzou`。
- 前端只调用本地管理 API，由 `internal/web/server.go` 中的路由接管，再做鉴权、限流、本地缓存、分片合并等处理。
- 所以“你要验证的 curl”分两层：
  1. 先调本地管理 API；
  2. 再由本地服务代你请求蓝奏云网页接口。

## 2. 蓝奏云侧默认地址

| 配置项 | 默认值 | 说明 |
|------|------|------|
| `BaseUrl` | `https://pc.woozooo.com` | 管理后台网页入口 |
| `ShareUrl` | `https://pan.lanzoui.com` | 分享页网页入口 |
| `RootFolderID` | `-1` | 默认根目录 |


### 5.1 登录与初始化

- `GET https://up.woozooo.com/mlogin.php`
- `POST https://up.woozooo.com/mlogin.php`
  - `task=3`
  - `uid=<账号>`
  - `pwd=<密码>`
- 成功后从响应 Cookie 中取登录态
- 如果返回 `acw_sc__v2`，会进入反爬校验

```bash

  -H 'Referer: https://pc.woozooo.com' \
  -H 'User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/142.0.0.0 Safari/537.36' \
  'https://up.woozooo.com/mlogin.php' > /tmp/lan_login_page.html

# 2. 提交登录；如果返回 acw_sc__v2，可继续按页面内容计算后重试
curl -s -b /tmp/lan_lanzou_cookie.txt \
  -X POST 'https://up.woozooo.com/mlogin.php' \
  -H 'Referer: https://pc.woozooo.com' \
```

### 5.6 下载地址解析

- 先在分享页解析参数/JS data，再请求 `ajaxm.php`
- 若没有直接 302，再请求 `baseUrl/ajax.php`
- 最终从响应或 302 Location 中得到真实下载地址
- 下载时通常带：
  - `Referer: https://pc.woozooo.com`
  - `Cookie: down_ip=1`
- 必要时附加 `acw_sc__v2`

```bash
# 1. 先请求下载地址页，观察是否 302
curl -s -b '/tmp/lan_lanzou_cookie.txt; down_ip=1' \
  -H 'Referer: https://pc.woozooo.com' \
  -H 'User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/142.0.0.0 Safari/537.36' \
  'https://pan.lanzoui.com/ajaxm.php?file=<fileID>' \
  -D /tmp/lan_dl_headers.txt

# 2. 若未直接 302，再请求 ajax.php
curl -s -b '/tmp/lan_lanzou_cookie.txt; down_ip=1' \
  -X POST '<baseUrl>/ajax.php' \
  -H 'Referer: https://pc.woozooo.com' \
  -H 'User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/142.0.0.0 Safari/537.36' \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  --data '<从分享页提取的 param，可补 el=2>'

# 3. 用最终下载地址下载
curl -L -b '/tmp/lan_lanzou_cookie.txt; down_ip=1' \
  -H 'Referer: https://pc.woozooo.com' \
  -H 'User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/142.0.0.0 Safari/537.36' \
  '<最终下载地址>'
```

### 5.7 反爬说明

这一节只解释“项目实际碰到的蓝奏云反爬逻辑”，不保证未来页面一更新还完全一样。

- 登录页和分享页都可能返回 `acw_sc__v2` 校验页；
- 下载链接请求通常要带 `Referer`；
- 下载相关请求通常要带 `down_ip=1`；
- 若触发了校验，页面里会出现 `arg1='...'`，需要先算出 `acw_sc__v2` 再重试。

对应逻辑在：

```text
internal/lanzou/help.go
internal/lanzou/util.go
internal/web/server.go
```

#### 5.7.1 本地通用请求里的反爬

这是项目里最核心的通用请求封装，很多蓝奏交互都会走到这里。

- 非 `/file/` 请求时，把 `acw_sc__v2` 挂到 `Cookie` 里；
- `/file/` 下载请求时，额外固定加 `down_ip=1`；
- 每次请求后检查响应体是否还含 `acw_sc__v2`；
- 若含校验页，就调用 `CalcAcwScV2` 计算新值并重试，最多 3 次。

```text
internal/lanzou/util.go:154
```

你手工 curl 时，如果要模仿这一层，注意两点：

- 不要把 `acw_sc__v2` 和其他 Cookie 分开写死成两段；这里代码里是合并后一起放进 `Cookie`；
- 下载类请求如果发现是 `/file/` 路径，会固定补 `down_ip=1`。

#### 5.7.2 `acw_sc__v2` 的计算逻辑

这是项目自己实现的校验值计算，不是标准公开文档里的写法。

- 先从 HTML 里提取 `arg1='XXXXXXXX...'`；
- 把 `arg1` 经过 `unbox` 重新排列；
- 优先按“2 字符一组 XOR”的方式计算；
- 也尝试页面里的其他长 hex 串作为密钥；
- 还保留旧版“整段 hexXor”兜底；
- 只要有一个候选非空，就当成新的 `acw_sc__v2`。

```text
internal/lanzou/help.go:121
```

所以如果你手工 curl 遇到这层，常见现象是：

- 第一次请求返回 HTML；
- HTML 里有 `arg1=`；
- HTML 里可能还有另一个长 hex；
- 这时不能直接拿业务 JSON，而要先处理校验。

常见做法是：

```bash
curl -s -b '/tmp/lan_lanzou_cookie.txt; down_ip=1' \
  -H 'Referer: https://pc.woozooo.com' \
  -H 'User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/142.0.0.0 Safari/537.36' \
  'https://pc.woozooo.com/doupload.php?uid=<uid>&vei=<vei>' \
  --data-urlencode 'task=5' \
  --data-urlencode 'folder_id=-1' \
  --data-urlencode 'pg=1' \
  -o /tmp/lan_antispy.html

grep -o "arg1='[^']*'" /tmp/lan_antispy.html | head -n 5
```

如果页面里还出现了其他长 hex，也可以一起保留下来，作为后续计算候选。

注意：项目代码里这里只是“算出来一个候选值就继续重试”，没有先人工判断哪个候选是对的。

#### 5.7.3 登录页的反爬

账号密码登录时，代码会先 `GET` 一次登录页预热；然后再 `POST` 登录表单。

这里有几个细节：

- 预热请求也走通用请求逻辑，所以也可能先碰到一次 `acw_sc__v2`；
- 登录表单本身会禁止自动跳转，靠 `loginRedirectSuccess` 判断是否成功；
- 如果返回的是 `acw_sc__v2` 页，就重新计算后继续重试，最多 5 次；
- 每次重试前会 `sleep 800ms`；
- 若最终仍失败，会提示“登录反爬验证失败，请改用 Cookie 登录”。

```text
internal/lanzou/util.go:316
```

你手工 curl 时，最容易在这里遇到两种情况：

- 返回了带 `arg1=` 的 HTML；
- 返回了非 JSON 页面，但状态码不是成功跳转。

这时候优先换成浏览器导出的 Cookie，比继续手算更稳。

#### 5.7.4 分享页/下载页的反爬

分享页和下载流程里也都会检查，而且不止检查一次。

- `getShareUrlHtml`：获取分享页时若发现 `acw_sc__v2`，就计算后重试，最多 3 次；
- `getFilesByShareUrl`：请求下载地址页时若未直接 302，也可能遇到 `acw_sc__v2`，于是转去请求 `ajax.php`；
- `ajax.php` 响应里若还有 `acw_sc__v2`，也会继续计算并重试，最多 3 次；
- 每次遇到这种情况，代码通常会 `sleep 2s` 再继续。

```text
internal/lanzou/util.go:476
internal/lanzou/util.go:618
internal/lanzou/util.go:665
```

#### 5.7.5 对 curl 验证的实际影响

如果你不是走本地管理 API，而是直接 curl 蓝奏网页接口，很容易遇到这些现象：

- 第一次请求返回 HTML，而不是 JSON；
- HTML 里有 `acw_sc__v2`、`arg1=`、`<script>`；
- 这时不能直接解析业务数据，而要先处理反爬页；
- 本地管理后台如果遇到这类错误，通常会缩短成“登录被反爬拦截，请改用 Cookie 登录”。

所以最稳的验证方式还是：

1. 先走本地管理 API；
2. 若一定要直接测蓝奏网页接口，优先用浏览器登录后导出的 Cookie；
3. 遇到 `acw_sc__v2` 时，先抓页面里的 `arg1` 和长 hex，再按项目逻辑重试。

#### 5.7.6 一个更完整的 curl 调试思路

如果你一定要手工追第 5 节里的蓝奏网页接口，下面这版会更偏向“抓包 + 对照字段位置”的调试方式。

推荐先看两类证据：

- curl 返回的响应体；

##### 5.7.6.5 最小可复现的 `acw_sc__v2` 抓包 checklist

这个 checklist 的目标不是“立刻算出最终值”，而是先把“是不是校验页、关键字段在哪里、下一步该复现哪一跳”这三点缩小到最小范围。

###### 1. 固定最小环境

- 用一个空 `Cookie` 文件开始，避免把浏览器里已经过期的 cookie 带进来：
  - `/tmp/lan_acw_cookie.txt`
- 固定用同一个 User-Agent，例如：
  - `Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/142.0.0.0 Safari/537.36`
- 固定 Referer：
  - 管理页用 `https://pc.woozooo.com`
  - 分享页用 `https://pan.lanzoui.com`

###### 2. 只保留最小一跳

先别一次把登录、列目录、下载全串起来。只保留最小一跳，比如：

- 管理页：`doupload.php?uid=...&vei=...` + `task=5`
- 分享页：`pan.lanzoui.com/<shareID>`
- 下载前页：`pan.lanzoui.com/ajaxm.php?file=...`

这样如果出问题，你能立刻知道是哪一跳触发的。

###### 3. 先拿原始响应，不要省掉诊断信息

推荐一开始就保存：

- 响应体到 `/tmp/lan_acw_body.html`
- 响应头到 `/tmp/lan_acw_headers.txt`

例如：

```bash
curl -s -b /tmp/lan_acw_cookie.txt   -H 'Referer: https://pc.woozooo.com'   -H 'User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/142.0.0.0 Safari/537.36'   'https://pc.woozooo.com/doupload.php?uid=<uid>&vei=<vei>'   --data-urlencode 'task=5'   --data-urlencode 'folder_id=-1'   --data-urlencode 'pg=1'   -o /tmp/lan_acw_body.html   -D /tmp/lan_acw_headers.txt
```

###### 4. 用最少命令判断是不是反爬页

先看这四件事：

- `grep -nE "acw_sc__v2|arg1=" /tmp/lan_acw_body.html | head -n 20`
- `grep -nE "\<script" /tmp/lan_acw_body.html | head -n 20`
- `grep -nE "Set-Cookie:" /tmp/lan_acw_headers.txt | head -n 20`
- `grep -oE "[0-9A-F]{32,}" /tmp/lan_acw_body.html | head -n 20`

如果命中，先不急着继续调接口，而是进入下一步：抓包对照。

###### 5. 抓包时先看这 5 个字段位置

在 Chrome DevTools 里找到最小复现的那条请求后，优先确认：

- 请求 URL 和 Method
- `Request Headers` 里的 `Cookie` 有没有 `acw_sc__v2`
- `Request Headers` 里的 `Referer`
- `Form Data` 或 `Query String` 里的参数
- `Response Headers` 里有没有 `Set-Cookie`

如果你发现浏览器请求带了 `acw_sc__v2`，但你 curl 没带，通常下一步就是补这个 cookie。

###### 6. 按“浏览器能成功，curl 1:1 复现”的顺序验证

最稳的顺序不是“直接硬算 `acw_sc__v2`”，而是：

- 先把浏览器里真实请求的 `curl` 大致拼出来；
- 再逐步删掉不必要字段，保留最小可复现集合；
- 若浏览器里需要带某个 JS 变量，再单独记下来。

参考对照：

| 浏览器 DevTools | 最小 curl 对照 |
| --- | --- |
| `General -> Request URL` | URL |
| `Headers -> Request Method` | `-X POST` / `GET` |
| `Form Data` | `--data-urlencode` / `-F` |
| `Response Headers -> Set-Cookie` | `-b /tmp/lan_acw_cookie.txt` |
| `Status Code` | 看是否 200 / 302 / 非 200 HTML |

###### 7. 记录最小可复现信息

每次复现时，建议至少保留：

- 触发接口 URL
- Method
- 是否带 `acw_sc__v2`
- 是否带 `down_ip=1`
- 是否返回 HTML
- 响应里有没有 `arg1=`
- 响应里有没有其他长 hex
- 这一步的结果：成功 JSON / 失败 HTML / 302

例如整理成：

```text
接口：https://pc.woozooo.com/doupload.php?uid=...&vei=...
Method：POST
Cookie：acw_sc__v2=...
Referer：https://pc.woozooo.com
结果：HTML
包含：acw_sc__v2 / arg1='...' / <script
下一步：提取 arg1 后重试 / 改用浏览器 Cookie / 回本地管理 API
```

###### 8. 常见最小复现顺序

如果你要快速确认“到底是哪一层在拦”，推荐按这个顺序试：

1. 分享页 `GET /<shareID>`
2. 分享密码页 `POST /ajaxm.php?file=...`
3. 管理页 `POST /doupload.php?uid=...&vei=...`
4. 上传页 `POST /html5up.php`
5. 下载前页 `POST /ajax.php`

一般前两步最容易直接命中 `acw_sc__v2`。

### 第一步：登录本地管理后台

```bash
curl -s -X POST "$BASE/api/auth/login" \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"<你的密码>"}' \
  -c /tmp/lan_cookie.txt
```

### 第二步：保存 Cookie 配置

把浏览器蓝奏登录后的 Cookie 贴到请求体里：

```bash
curl -s -X POST "$BASE/api/config/save" \
  -b /tmp/lan_cookie.txt \
  -H 'Content-Type: application/json' \
  -d '{"type":"cookie","cookie":"<浏览器Cookie>","base_url":"https://pc.woozooo.com","share_url":"https://pan.lanzoui.com","root_folder_id":"-1"}'
```

### 第三步：验证蓝奏连接

```bash
curl -s "$BASE/api/status" -b /tmp/lan_cookie.txt
```

### 第四步：列目录验证

```bash
curl -s "$BASE/api/files?folder_id=-1" -b /tmp/lan_cookie.txt | jq .
```

### 第五步：分享验证

```bash
curl -s -X POST "$BASE/api/files/share" \
  -b /tmp/lan_cookie.txt \
  -H 'Content-Type: application/json' \
  -d '{"id":"<文件ID>","is_dir":false,"enable_password":false,"password":""}' | jq .
```

### 第六步：直链验证

